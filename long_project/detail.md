## 7.3 사용자 시나리오 (정상 흐름 + 예외 흐름)

### 시나리오: [예: 게시글 작성]
**정상 흐름**
1. 사용자가 로그인 상태에서 "글쓰기" 버튼 클릭
2. 작성 화면으로 이동
3. 제목/내용 입력 후 "등록" 클릭
4. 게시글 목록으로 이동, 새 글이 최상단에 표시

**예외 흐름**
- 제목/내용 미입력 시 → [에러 메시지 노출, 등록 버튼 비활성화 등]

- 네트워크 오류 시 → [재시도 안내]

# 8. ERD 설계 노트

> ERD는 실제 도식(ERDCloud, dbdiagram.io 등 툴 사용 추천)으로 그리고, 아래 표는 그리기 전 정리용으로 사용하세요.

## 8.1 엔티티 목록
| 엔티티명 | 설명 |
|---|---|
| users | 사용자 정보 |
| posts | 게시글 |
| comments | 댓글 |

## 8.2 엔티티별 컬럼 정의

### users
| 컬럼명 | 타입 | 제약조건 | 설명 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 회원 고유 ID |
| email | VARCHAR(255) | UNIQUE, NOT NULL | 이메일 |
| password | VARCHAR(255) | NOT NULL | 암호화된 비밀번호 |
| nickname | VARCHAR(50) | NOT NULL | 닉네임 |
| created_at | DATETIME | NOT NULL | 생성일시 |
| deleted_at | DATETIME | NULL | 소프트 삭제 일시 |

### posts
| 컬럼명 | 타입 | 제약조건 | 설명 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 게시글 ID |
| user_id | BIGINT | FK → users.id | 작성자 |
| title | VARCHAR(255) | NOT NULL | 제목 |
| content | TEXT | NOT NULL | 내용 |
| created_at | DATETIME | NOT NULL | 작성일시 |

## 8.3 관계 정의
| 관계 | 설명 | 카디널리티 |
|---|---|---|
| users - posts | 한 사용자는 여러 게시글 작성 | 1:N |
| posts - comments | 한 게시글에 여러 댓글 | 1:N |
| users - comments | 한 사용자는 여러 댓글 작성 | 1:N |


## 9.2 API 목록

### [회원] 회원가입
| 항목 | 내용 |
|---|---|
| URL | `POST /api/users/signup` |
| 요청 헤더 | 없음 |
| 요청 Body | `{ "email": "string", "password": "string", "nickname": "string" }` |
| 응답 (200) | `{ "success": true, "data": { "id": 1, "email": "...", "nickname": "..." } }` |
| 예외 처리 | 이메일 중복(409), 형식 오류(400) |

### [회원] 로그인
| 항목 | 내용 |
|---|---|
| URL | `POST /api/users/login` |
| 요청 Body | `{ "email": "string", "password": "string" }` |
| 응답 (200) | `{ "success": true, "data": { "accessToken": "..." } }` |
| 예외 처리 | 이메일/비밀번호 불일치(401) |

### [게시글] 목록 조회
| 항목 | 내용 |
|---|---|
| URL | `GET /api/posts?page=1&size=10` |
| 요청 헤더 | 선택 (비로그인 가능) |
| 응답 (200) | `{ "success": true, "data": { "posts": [...], "totalCount": 100 } }` |
| 예외 처리 | 없음 |

> 💡 위 형식을 복사해서 요구사항 명세서(3번)의 기능 하나당 API 1개 이상 작성하세요.
